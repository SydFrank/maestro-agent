# ADR-001：Agent 编排框架选型

- 状态：已采纳
- 日期：2026-06-27
- 决策者：项目负责人
- 相关：[[ADR-002-reasoning-paradigm]]、[[ADR-005-multi-agent-topology]]

## 背景

FinSight 是一个多 Agent 投研平台，核心是"编排多个专职 Agent 协作完成一次投研任务"。
编排框架是地基，决定了：可控性、可观测性、能否做断点/回放/人审、以及未来演进成本。
错误的地基会让后续所有努力打折，因此单列一份决策记录。

约束与目标（按优先级）：
1. **可控、可追溯**——投研场景对结果可解释、可审计要求极高（金融不能黑盒）。
2. **支持多 Agent 协作 + 有界循环**——必须能精确控制"谁在什么时候做什么"。
3. **可观测**——每一步推理、工具调用、token/成本都要能被记录与回放。
4. **多模型可切换**——主 Claude Opus 4.8，备 OpenAI，框架不应绑死单一厂商。
5. **演进空间**——后续要能加 checkpoint、人工审批(HITL)、并行执行。

## 候选方案

| 框架 | 范式 | 优势 | 关键劣势（为什么不选） |
|---|---|---|---|
| **LangGraph**（采纳）| 显式状态图 | 循环/分支/回退全可控；子图可嵌套做分层多 Agent；原生支持 checkpoint、中断、并行 | API 偏底层、需自己设计 state（可接受，换来可控性）|
| LangChain AgentExecutor | 封装好的 Agent 循环 | 上手最快 | **循环是黑盒**，出错难定位、难加自定义控制——与"可控可追溯"硬冲突 |
| CrewAI | 角色制"团队" | Demo 极快、概念直观 | 封装过死，企业级定制、细粒度可观测、自定义编排吃力 |
| AutoGen / AG2 | 对话驱动多 Agent | 多 Agent 灵活、研究友好 | 对话式协作**难控制终止、易跑偏**，生产可预测性差 |
| LlamaIndex | RAG 为中心 | RAG 生态最强 | Agent 编排能力弱，复杂多 Agent 流程吃力（其 RAG 能力可单独借鉴） |
| Claude Agent SDK | Claude 原生 Agent | 与 Claude 配合最佳、工程化强 | **绑定单厂商**，与"多模型可切换"目标冲突（但其 harness 设计值得学习与对标）|
| 自研循环 | 完全自定义 | 100% 可控 | 重复造轮子：checkpoint、并行、回放都要自己实现，成本过高 |

## 决策

**采用 LangGraph 作为多 Agent 编排框架。**

理由：它是唯一同时满足"显式可控 + 可追溯 + 支持分层多 Agent + 原生 checkpoint/中断/并行 + 不绑模型厂商"的框架。
我们把每个专职工人建成一个有界 ReAct 子流程，再用一张 Supervisor 主图编排它们——
整个系统始终是一张**可视化、可回放、可插入人审**的状态图。

## 后果

**正面：**
- 推理过程对前端的"轨迹可视化/评测平台"天然友好（Harness Engineering 的基础）。
- 加 HITL、并行 fan-out、断点续跑都是框架原生能力，演进成本低。
- 与模型解耦：模型调用统一走 llm-gateway，换 Claude/OpenAI 无需改图。

**负面 / 需主动管理的风险：**
- state 设计不当会变乱 → 用 `TypedDict` 显式定义、分字段存（工具结果/引用/用量分开）。
- 偏底层、样板代码多 → 用共享的 worker 抽象封装重复逻辑。

**被否方案的保留价值：**
- LlamaIndex 的检索策略、Claude Agent SDK 的 harness 设计，作为对标与借鉴对象，写入后续 ADR。

## 面试追问预案（自检）

- Q：为什么不用 CrewAI/AutoGen，它们不是更适合多 Agent 吗？
  A：它们适合**快速搭原型**，但企业要**可预测、可追溯**。AutoGen 的对话式协作终止难控、路径不可复现；CrewAI 封装太死，细粒度可观测和自定义编排做不了。LangGraph 用"显式状态图"换来了这两样。
- Q：为什么不用 Claude Agent SDK，你不是主用 Claude 吗？
  A：模型用 Claude，但框架绑死厂商会牺牲"多模型可切换"。我把模型抽象在 llm-gateway，框架层保持中立——这是分层解耦。Claude Agent SDK 的 harness 设计我有研究，作为对标。
