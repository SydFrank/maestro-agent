# RAG 策略决策卡（详细对比）

> 配合 [architecture-decisions.md](architecture-decisions.md) 的 ADR-08~11 阅读。
> 每张卡：候选 · 选谁 · 机制为什么好 · 达到的效果 · 备选缺点 · 面试话术。
>
> **元原则**：没有"最好的策略"，只有在延迟/成本/质量约束下最优的。面试官问"为什么选 A"，考的是你懂权衡，不是背答案。

---

## 卡 1：检索方式 — Hybrid vs 纯向量 vs 纯 BM25

- **候选**：A=Hybrid(向量+BM25, RRF 融合) · B=纯 Dense 向量 · C=纯 Sparse(BM25)
- **选 A**
- **机制**：向量擅长语义（"如何请假"→"休假申请流程"），但对精确词弱（型号 X-2000、错误码 ERR_507、人名）；BM25 正相反。两者召回不同类型的相关文档，RRF 倒数排名融合即互补。
- **效果**：recall@k 显著提升，尤其在含专名/代码/编号的企业文档上。
- **B 缺点**：纯向量漏精确匹配，对 OOV 新术语几乎无能为力。
- **C 缺点**：纯 BM25 不懂同义改写，换个说法就找不到。
- **话术**："企业文档大量型号/编号/专名，纯向量召回差，所以 Hybrid + RRF 让语义和关键词互补。"

## 卡 2：要不要 Rerank

- **候选**：A=Cross-encoder 重排 · B=不重排 · C=只用 MMR 去重
- **选 A**
- **机制**：第一阶段 bi-encoder（query/doc 分开编码，可预存）快但精度有限；Rerank 用 cross-encoder（query+doc 拼一起算真实相关性）慢但准。两阶段：快召回 top-50 → 准精排 top-5。
- **效果**：送入上下文的片段更准 → **直接降幻觉**（料准了模型瞎编空间小），并缓解 lost-in-the-middle。
- **B 缺点**：top-k 混入"看着像但不相关"的噪声，带偏模型 → 幻觉。
- **C 缺点**：MMR 只解决重复，不提升相关性精度，治标不治本。
- **话术**："召回/精排分两阶段，因为单靠向量相似度 top-k 噪声多，而噪声是幻觉主因。"

## 卡 3：分块 — Contextual Retrieval vs 朴素切块 vs 大块

- **候选**：A=Contextual Retrieval(块前注入上下文摘要) · B=朴素定长切块 · C=整文档不切
- **选 A**（Anthropic 提出的技术）
- **机制**：朴素切块的病是"块脱离上下文"——块写"该比例上升 3%"，但"该比例/哪家/哪季"在别的块。Contextual Retrieval 在 embed 前用 LLM 给每块加一句"它在全文里讲什么"的前缀，再 embed，让块自带身份。
- **效果**：Anthropic 基准里 Contextual Embedding 降 top-20 检索失败率约 35%，叠加 Contextual BM25 约 49%，再加 Rerank 约 67%。**注意：随语料变化，面试说"方向+机制"，数字以自己评测集为准。**
- **B 缺点**：块语义残缺、召回错位，是朴素 RAG 质量差的头号原因。
- **C 缺点**：粒度太粗、塞爆窗口、成本高、触发 lost-in-the-middle。
- **代价与对策**：摄取要逐块调 LLM → 便宜模型 + prompt caching（缓存整篇，逐块复用）摊薄。
- **话术**："朴素切块核心病是块脱离上下文，我用 Contextual Retrieval 给每块加 LLM 上下文摘要——Anthropic 验证过能显著降检索失败率。"

## 卡 4：Agentic RAG vs 静态 RAG 管道

- **候选**：A=Agentic RAG(Agent 自决何时/检索什么) · B=静态管道(永远先检索再答) · C=不检索全靠模型
- **选 A**（本项目 Knowledge 工人即是）
- **机制**：静态管道无脑检索（"你好"也查库 → 浪费+噪声）。Agentic RAG 让 Agent 判断要不要检索、用什么 query、够不够再检一轮。
- **效果**：该检索时检索（准），不该时不检索（省+少噪声），多跳问题多轮覆盖。
- **B 缺点**：闲聊也检索浪费成本+噪声污染；复杂问题一次不够答不全。
- **C 缺点**：纯参数知识答不了私有知识 + 幻觉拉满。
- **话术**："Knowledge 工人是 Agentic RAG，检索时机和 query 由 Agent 决策，不是死板的先检索再答。"

## 卡 5：查询改写 — 何时用，何时别用

- **候选**：A=直接用原 query · B=Multi-query · C=HyDE(先让 LLM 编假答案再检索)
- **选 A 为默认，B/C 按需**（重点是不滥用）
- **机制**：B/C 都能提召回，但都多一次 LLM 调用+延迟。Multi-query 适合"问题模糊/多义"，HyDE 适合"长尾、query 与文档措辞差异大"。
- **为什么默认 A**：清晰查询 + Hybrid + Rerank 已够好。无脑上 HyDE 是典型过度工程——多花钱多延迟，提升不明显，还可能因"假答案编错方向"反噬召回。
- **B/C 缺点**：成本/延迟翻倍；HyDE 在陌生领域编错假设反噬召回。
- **话术**："查询改写按需用不默认开，因为 Hybrid+Rerank 已覆盖大部分，只在召回不达标的长尾才划算——评测数据驱动，不堆技术。"

## 关于"达到什么效果"的硬规矩

> **你说出口的数字，必须是你自己评测跑出来的。** 背"提升 35%"很危险——一问"怎么测的、什么数据集、baseline 是啥"就穿。

正确姿势：**讲机制（为什么有效）+ 讲方向（提召回/降幻觉）+ 用自己评测集的数**。这是项目必须配评测集的原因——它把"我觉得"变成"我测过"。
